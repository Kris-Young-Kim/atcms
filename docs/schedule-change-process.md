# 일정 변경 프로세스

**프로젝트 코드**: ATCMP-2026  
**작성일**: 2025-01-27  
**버전**: 1.0

---

## 📋 개요

이 문서는 프로젝트 일정 변경을 체계적으로 관리하기 위한 프로세스를 정의합니다. 일정 변경의 영향도를 분석하고 적절한 승인을 받아 프로젝트의 일정을 효과적으로 관리합니다.

---

## 🎯 목적

1. **일정 변경의 체계적 관리**: 모든 일정 변경이 문서화되고 추적됨
2. **영향도 기반 승인**: 일정 변경의 영향도에 따라 적절한 승인 프로세스 적용
3. **이해관계자 커뮤니케이션**: 일정 변경 시 모든 이해관계자에게 적시에 통보
4. **일정 관리 개선**: 일정 변경의 원인 분석을 통한 프로세스 개선

---

## 📝 일정 변경 유형

### 1. 스프린트 일정 조정

**정의**: 현재 스프린트 내 작업 조정

**예시**:
- 작업 우선순위 변경
- 작업 재할당
- 스프린트 목표 조정

**영향도**: 낮음

---

### 2. 마일스톤 조정

**정의**: 프로젝트 마일스톤 일정 변경

**예시**:
- 마일스톤 연기
- 마일스톤 당기기
- 마일스톤 내용 변경

**영향도**: 중간 ~ 높음

---

### 3. 릴리즈 일정 변경

**정의**: 프로젝트 릴리즈 일정 변경

**예시**:
- 릴리즈 연기
- 릴리즈 당기기
- 릴리즈 범위 변경

**영향도**: 높음

---

### 4. 휴가/부재 일정 반영

**정의**: 팀원 휴가 또는 부재로 인한 일정 조정

**예시**:
- 장기 휴가 일정 반영
- 병가/특별 휴가 반영
- 교육/회의 참석으로 인한 부재

**영향도**: 낮음 ~ 중간

---

## 🔄 일정 변경 프로세스

### 단계 1: 일정 변경 요청 제출

1. **일정 변경 이슈 생성**
   - GitHub Issue 생성
   - 일정 변경 사유 명확히 기록
   - 필수 정보 입력:
     - 변경 유형
     - 변경 전 일정
     - 변경 후 일정
     - 변경 사유
     - 영향 받는 작업/마일스톤

2. **라벨 설정**
   - `type: schedule-change` 라벨 추가
   - 우선순위 라벨 설정 (`priority: P0-P3`)
   - 영향도 라벨 설정 (`impact: low/medium/high`)

---

### 단계 2: 영향도 분석

1. **작업 영향도 분석**
   - 영향 받는 작업 식별
   - 작업 간 의존성 확인
   - 작업 일정 재계획

2. **리소스 영향도 분석**
   - 필요한 개발자 리소스 확인
   - 리소스 가용성 확인
   - 리소스 재할당 필요성 평가

3. **이해관계자 영향도 분석**
   - 영향 받는 이해관계자 식별
   - 영향 받는 외부 프로젝트/팀 확인
   - 커뮤니케이션 필요성 평가

4. **비용 영향도 분석**
   - 일정 변경으로 인한 비용 증가
   - 기회 비용 평가
   - 예산 영향 평가

**분석 결과**:
- **낮음**: 스프린트 내 조정 가능, 외부 영향 없음
- **중간**: 마일스톤 조정 필요, 일부 이해관계자 영향
- **높음**: 릴리즈 일정 변경 필요, 전체 이해관계자 영향

---

### 단계 3: 승인 프로세스

#### 낮은 영향도 변경 (스프린트 내 조정)

**승인 주체**: PM

**프로세스**:
1. PM 검토
2. 개발팀과 간단히 논의
3. 승인/거부 결정
4. 승인 시 즉시 일정 조정

**목표 처리 시간**: 1일 이내

---

#### 중간 영향도 변경 (마일스톤 조정 < 1주)

**승인 주체**: PM + 기술 리더

**프로세스**:
1. PM 및 기술 리더 검토
2. 영향도 분석 문서 작성
3. 개발팀 회의
4. 승인/거부 결정
5. 승인 시 이해관계자 통보

**목표 처리 시간**: 3일 이내

---

#### 높은 영향도 변경 (마일스톤 조정 > 1주 또는 릴리즈 변경)

**승인 주체**: PM + 기술 리더 + 프로젝트 스폰서

**프로세스**:
1. 상세 영향도 분석 문서 작성
2. 이해관계자 회의 소집
3. 대안 검토 및 토론
4. 승인/거부 결정
5. 승인 시 프로젝트 계획 수정 및 공식 통보

**목표 처리 시간**: 1주 이내

---

### 단계 4: 일정 변경 실행

1. **일정 업데이트**
   - 프로젝트 일정 문서 업데이트
   - GitHub 프로젝트 보드 업데이트
   - 마일스톤 일정 업데이트
   - 캘린더 업데이트

2. **이해관계자 통보**
   - 변경 일정 공지
   - 변경 사유 설명
   - 영향 받는 작업/마일스톤 공지
   - 새로운 일정 공지

3. **작업 재계획**
   - 작업 우선순위 재조정
   - 작업 할당 재조정
   - 리소스 재할당

---

### 단계 5: 일정 변경 완료 및 검토

1. **변경 사항 확인**
   - 모든 일정 문서 업데이트 확인
   - 이해관계자 통보 완료 확인
   - 작업 재계획 완료 확인

2. **일정 변경 종료**
   - 이슈 상태를 "완료"로 변경
   - 완료 일자 기록
   - 변경 효과 평가

3. **회고**
   - 일정 변경 원인 분석
   - 일정 변경 프로세스 개선점 파악
   - 학습 내용 공유

---

## 👥 역할 및 책임

### 일정 변경 요청자

- 일정 변경 이슈 작성
- 변경 사유 및 영향도 설명
- 영향도 분석 지원
- 승인 후 일정 업데이트

---

### PM (Project Manager)

- 일정 변경 요청 검토
- 영향도 분석 주도
- 승인 프로세스 관리
- 이해관계자 커뮤니케이션
- 일정 문서 업데이트

---

### 기술 리더

- 기술적 영향도 평가
- 리소스 가용성 확인
- 개발팀 일정 조정 지원

---

### 프로젝트 스폰서

- 높은 영향도 일정 변경 승인
- 이해관계자 커뮤니케이션 지원
- 예산/리소스 승인

---

## 📊 일정 변경 추적

### 추적 지표

- **일정 변경 수**: 월별 일정 변경 요청 수
- **승인률**: 승인된 일정 변경 비율
- **처리 시간**: 일정 변경 요청부터 승인까지 평균 시간
- **일정 연기율**: 일정 연기 비율
- **일정 준수율**: 원래 일정 대비 준수율

### 리뷰 주기

- **주간**: 진행 중인 일정 변경 상태 확인
- **월간**: 일정 변경 통계 및 트렌드 분석
- **분기별**: 일정 변경 프로세스 개선 검토

---

## 🚫 일정 변경 거부 기준

다음의 경우 일정 변경을 거부할 수 있습니다:

1. **프로젝트 목표와 불일치**: 일정 변경이 프로젝트 목표 달성에 부정적 영향
2. **이해관계자 승인 불가**: 주요 이해관계자가 일정 변경 승인 불가
3. **예산 제약**: 일정 변경으로 인한 비용 증가 승인 불가
4. **법적/계약상 제약**: 계약상 일정 변경 불가
5. **위험도 과다**: 일정 변경으로 인한 위험이 과도함

거부 시 반드시 이유를 명확히 문서화해야 합니다.

## 📅 일정 변경 통보 체계

### 통보 시점

- **낮은 영향도**: 개발팀 내 통보 (승인 후 즉시)
- **중간 영향도**: 관련 이해관계자 통보 (승인 후 1일 이내)
- **높은 영향도**: 전체 이해관계자 공식 통보 (승인 후 즉시)

### 통보 방법

- **이메일**: 공식 통보
- **Slack**: 빠른 공지
- **회의**: 중요한 변경 시 회의로 통보
- **문서**: 일정 변경 문서 업데이트

### 통보 내용

- 변경 전 일정
- 변경 후 일정
- 변경 사유
- 영향 받는 작업/마일스톤
- 새로운 일정에 대한 설명
