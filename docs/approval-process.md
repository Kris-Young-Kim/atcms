# 승인 프로세스 가이드

**프로젝트 코드**: ATCMP-2026  
**작성일**: 2025-01-27  
**버전**: 1.0

---

## 📋 개요

이 문서는 프로젝트에서 발생하는 다양한 승인 요청을 체계적으로 관리하기 위한 프로세스를 정의합니다. 승인 권한 매트릭스를 기반으로 적절한 승인자에게 승인을 요청하고 승인 이력을 추적합니다.

---

## 🎯 목적

1. **승인 권한 명확화**: 승인 권한과 책임이 명확히 정의됨
2. **승인 프로세스 표준화**: 일관된 방식으로 승인 프로세스 진행
3. **승인 이력 추적**: 모든 승인이 문서화되고 추적됨
4. **승인 효율화**: 승인 시간 목표를 설정하여 효율적 승인

---

## 📝 승인 유형

### 1. 예산 승인

**정의**: 프로젝트 예산 사용 및 변경 승인

**예시**:
- 인프라 비용 승인
- 외부 서비스 구독 승인
- 도구 구매 승인
- 예산 초과 승인

---

### 2. 리소스 승인

**정의**: 인력 및 외부 리소스 사용 승인

**예시**:
- 추가 개발자 채용 승인
- 외부 컨설턴트 계약 승인
- 교육/세미나 참석 승인

---

### 3. 기술 승인

**정의**: 기술 스택 및 아키텍처 변경 승인

**예시**:
- 새로운 라이브러리 도입 승인
- 아키텍처 변경 승인
- 보안 정책 변경 승인

---

### 4. 프로덕션 배포 승인

**정의**: 프로덕션 환경 배포 승인

**예시**:
- 프로덕션 배포 승인
- 핫픽스 배포 승인
- 주요 릴리즈 배포 승인

---

## 👥 승인 권한 매트릭스

| 승인 항목 | 금액/영향 범위 | 승인 주체 | 승인 시간 목표 |
|---------|--------------|----------|---------------|
| **예산** | < 100만원 | PM | 1일 |
| **예산** | 100만원 ~ 500만원 | PM + 프로젝트 스폰서 | 3일 |
| **예산** | > 500만원 | 프로젝트 스폰서 + 재무 담당자 | 5일 |
| **리소스** | 추가 개발자 1명 | PM + 기술 리더 | 3일 |
| **리소스** | 추가 개발자 2명 이상 | PM + 프로젝트 스폰서 | 5일 |
| **기술** | 작은 변경 (< 1일) | 기술 리더 | 1일 |
| **기술** | 중간 변경 (1-3일) | 기술 리더 + PM | 3일 |
| **기술** | 큰 변경 (> 3일) | 기술 리더 + PM + 프로젝트 스폰서 | 5일 |
| **프로덕션 배포** | 일반 배포 | 기술 리더 + PM | 1일 |
| **프로덕션 배포** | 주요 릴리즈 | 기술 리더 + PM + 프로젝트 스폰서 | 2일 |

## 🔄 승인 프로세스

### 단계 1: 승인 요청 제출

1. **승인 요청 문서 작성**
   - 승인 요청 양식 작성
   - 승인 항목 상세 설명
   - 승인 사유 및 배경 설명
   - 예상 비용/영향도 포함

2. **승인 요청 이슈 생성**
   - GitHub Issue 생성 (기술 승인)
   - 또는 Notion 페이지 생성 (예산/리소스 승인)
   - 승인 권한 매트릭스에 따라 승인자 지정

3. **승인자에게 통보**
   - 승인자에게 승인 요청 통보
   - 승인 마감일 안내
   - 승인 필요 정보 제공

---

### 단계 2: 승인 검토

1. **승인자 검토**
   - 승인 요청 문서 검토
   - 필요 정보 확인
   - 추가 정보 요청 (필요시)

2. **의견 수집**
   - 관련 이해관계자 의견 수집
   - 기술 검토 (기술 승인)
   - 재무 검토 (예산 승인)

3. **승인 준비**
   - 승인/거부 결정 준비
   - 승인 조건 확인

---

### 단계 3: 승인 결정

1. **승인/거부 결정**
   - 승인자 결정
   - 승인 조건 명시 (필요시)
   - 거부 시 이유 명시

2. **승인 문서화**
   - 승인 상태 기록
   - 승인 일자 기록
   - 승인자 서명 (필요시)

3. **승인자 통보**
   - 승인 요청자에게 승인 결과 통보
   - 승인 조건 안내 (필요시)

---

### 단계 4: 승인 실행

1. **승인 조건 확인**
   - 승인 조건 충족 확인
   - 필요 문서 준비

2. **승인 항목 실행**
   - 승인된 항목 실행
   - 진행 상황 추적

3. **실행 결과 보고**
   - 승인 항목 실행 완료 보고
   - 결과 및 효과 평가

---

### 단계 5: 승인 완료 및 검토

1. **승인 항목 완료 확인**
   - 승인 항목 실행 완료 확인
   - 목표 달성 여부 확인

2. **승인 이력 업데이트**
   - 승인 상태를 "완료"로 변경
   - 완료 일자 기록
   - 결과 요약

3. **회고**
   - 승인 프로세스 개선점 파악
   - 승인 시간 목표 달성 여부 확인

---

## ⏱️ 승인 시간 목표

### 승인 시간 계산

**승인 시간** = 승인 요청일 ~ 승인 결정일

### 승인 시간 목표

- **긴급 승인**: 4시간 이내
- **일반 승인**: 승인 권한 매트릭스에 명시된 목표 시간
- **복잡한 승인**: 5일 이내 (추가 검토 필요 시)

### 승인 지연 처리

- 승인 시간 목표 초과 시:
  1. 승인자에게 리마인더 발송
  2. 승인자 상급자에게 알림 (필요시)
  3. 승인 지연 이유 기록

---

## 📊 승인 추적

### 추적 지표

- **승인 요청 수**: 월별 승인 요청 수
- **승인률**: 승인된 요청 비율
- **승인 시간**: 승인 요청부터 승인까지 평균 시간
- **승인 지연율**: 승인 시간 목표 초과 비율

### 리뷰 주기

- **주간**: 진행 중인 승인 요청 상태 확인
- **월간**: 승인 통계 및 트렌드 분석
- **분기별**: 승인 프로세스 개선 검토

---

## 🚫 승인 거부 기준

다음의 경우 승인을 거부할 수 있습니다:

1. **예산 초과**: 예산 범위를 초과하는 요청
2. **프로젝트 목표와 불일치**: 프로젝트 목표와 맞지 않는 요청
3. **위험도 과다**: 요청으로 인한 위험이 과도함
4. **리소스 부족**: 필요한 리소스가 부족함
5. **정책 위반**: 조직 정책 또는 보안 정책 위반

거부 시 반드시 이유를 명확히 문서화해야 합니다.

- **기술 승인**: GitHub Issues

- 승인 사유
- 승인 상태
- 승인 일자
- 승인자
- 승인 조건 (필요시)
- 거부 사유 (거부 시)
