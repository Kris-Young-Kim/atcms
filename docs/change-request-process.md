# 요구사항 변경 프로세스

**프로젝트 코드**: ATCMP-2026  
**작성일**: 2025-01-27  
**버전**: 1.0

---

## 📋 개요

이 문서는 프로젝트 진행 중 발생하는 요구사항 변경을 체계적으로 관리하기 위한 프로세스를 정의합니다. 변경 요청의 영향도를 평가하고 적절한 승인을 받아 프로젝트의 안정성을 유지합니다.

---

## 🎯 목적

1. **변경 요청의 체계적 관리**: 모든 변경 요청이 문서화되고 추적됨
2. **영향도 기반 승인**: 변경의 영향도에 따라 적절한 승인 프로세스 적용
3. **일정 관리**: 변경 요청으로 인한 일정 영향 최소화
4. **투명성**: 변경 요청의 배경과 이유가 명확히 문서화됨

---

## 📝 변경 요청 유형

### 1. 기능 추가 (Feature Addition)

**정의**: 새로운 기능을 추가하는 요청

**예시**:
- 새로운 사용자 스토리 추가
- 새로운 API 엔드포인트 추가
- 새로운 UI 컴포넌트 추가

**영향도**: 중간 ~ 높음

---

### 2. 기능 수정 (Feature Modification)

**정의**: 기존 기능을 수정하는 요청

**예시**:
- 기존 기능의 동작 변경
- UI/UX 개선
- 성능 최적화

**영향도**: 낮음 ~ 중간

---

### 3. 기능 제거 (Feature Removal)

**정의**: 기존 기능을 제거하는 요청

**예시**:
- 더 이상 필요 없는 기능 제거
- 사용되지 않는 API 제거

**영향도**: 낮음 ~ 중간

---

### 4. 우선순위 변경 (Priority Change)

**정의**: 기존 작업의 우선순위를 변경하는 요청

**예시**:
- 스프린트 내 작업 우선순위 변경
- 백로그 항목 우선순위 조정

**영향도**: 낮음

---

## 🔄 변경 요청 프로세스

### 단계 1: 변경 요청 제출

1. **변경 요청 이슈 생성**
   - GitHub Issue 생성
   - [변경 요청 템플릿](../.github/ISSUE_TEMPLATE/change_request.md) 사용
   - 필수 정보 입력:
     - 변경 요청 유형
     - 변경 내용 상세 설명
     - 변경 이유
     - 예상 영향도

2. **라벨 설정**
   - `type: change-request` 라벨 추가
   - 우선순위 라벨 설정 (`priority: P0-P3`)
   - 영향도 라벨 설정 (`impact: low/medium/high`)

---

### 단계 2: 영향도 평가

1. **기술적 영향도 평가**
   - 영향 받는 모듈/컴포넌트 식별
   - 예상 작업량 추정
   - 기술적 복잡도 평가
   - 테스트 범위 평가

2. **일정 영향도 평가**
   - 현재 스프린트 일정 영향
   - 마일스톤 일정 영향
   - 릴리즈 일정 영향

3. **리소스 영향도 평가**
   - 필요한 개발자 수
   - 필요한 작업 시간
   - 필요한 외부 리소스

4. **위험도 평가**
   - 기술적 위험
   - 일정 위험
   - 품질 위험

**평가 결과**:
- **낮음**: 작업량 < 1일, 일정 영향 없음
- **중간**: 작업량 1-3일, 스프린트 내 조정 가능
- **높음**: 작업량 > 3일, 마일스톤/릴리즈 영향

---

### 단계 3: 승인 프로세스

#### 낮은 영향도 변경 (< 1일)

**승인 주체**: PM

**프로세스**:
1. PM 검토
2. 기술 리더 간단 확인 (선택사항)
3. 승인/거부 결정
4. 승인 시 즉시 실행 가능

**목표 처리 시간**: 1일 이내

---

#### 중간 영향도 변경 (1-3일)

**승인 주체**: PM + 기술 리더

**프로세스**:
1. PM 및 기술 리더 검토
2. 영향도 분석 문서 작성
3. 관련 개발자 의견 수집
4. 승인/거부 결정
5. 승인 시 다음 스프린트에 포함

**목표 처리 시간**: 3일 이내

---

#### 높은 영향도 변경 (> 3일)

**승인 주체**: PM + 기술 리더 + 프로젝트 스폰서

**프로세스**:
1. 상세 영향도 분석 문서 작성
2. 이해관계자 회의 소집
3. 대안 검토 및 토론
4. 승인/거부 결정
5. 승인 시 프로젝트 계획 수정 필요

**목표 처리 시간**: 1주 이내

---

### 단계 4: 변경 실행

1. **작업 항목 생성**
   - 변경 요청을 작업 항목으로 분해
   - 스토리 포인트 추정
   - 담당자 할당
   - 일정 설정

2. **스프린트 계획 업데이트**
   - 스프린트 백로그에 추가
   - 우선순위 조정
   - 일정 재계획 (필요시)

3. **실행 및 추적**
   - 작업 진행 상황 추적
   - 정기적 업데이트
   - 완료 후 검증

---

### 단계 5: 변경 완료 및 검토

1. **변경 사항 검증**
   - 요구사항 충족 확인
   - 테스트 완료 확인
   - 문서 업데이트 확인

2. **변경 요청 종료**
   - 이슈 상태를 "완료"로 변경
   - 완료 일자 기록
   - 변경 효과 평가

3. **회고**
   - 변경 요청 프로세스 개선점 파악
   - 학습 내용 공유

---

## 👥 역할 및 책임

### 변경 요청 제출자

- 변경 요청 이슈 작성
- 변경 내용 및 이유 설명
- 영향도 평가 지원
- 승인 후 작업 실행

---

### PM (Project Manager)

- 변경 요청 검토
- 영향도 평가 주도
- 승인 프로세스 관리
- 일정 조정
- 이해관계자 커뮤니케이션

---

### 기술 리더

- 기술적 영향도 평가
- 기술적 실행 가능성 검토
- 기술적 위험 평가
- 개발자 리소스 확인

---

### 개발자

- 기술적 영향도 분석 지원
- 작업량 추정
- 실행 가능성 검토
- 작업 실행

---

## 📊 변경 요청 추적

### 추적 지표

- **변경 요청 수**: 월별 변경 요청 수
- **승인률**: 승인된 변경 요청 비율
- **처리 시간**: 변경 요청 제출부터 승인까지 평균 시간
- **일정 영향**: 변경 요청으로 인한 일정 연기 건수

### 리뷰 주기

- **주간**: 진행 중인 변경 요청 상태 확인
- **월간**: 변경 요청 통계 및 트렌드 분석
- **분기별**: 변경 요청 프로세스 개선 검토

---

## 🚫 변경 요청 거부 기준

다음의 경우 변경 요청을 거부할 수 있습니다:

1. **프로젝트 목표와 불일치**: 변경 요청이 프로젝트 목표와 맞지 않음
2. **기술적 불가능**: 현재 기술 스택으로 구현 불가능
3. **일정 영향 과다**: 릴리즈 일정에 심각한 영향
4. **예산 초과**: 추가 예산이 필요한데 승인되지 않음
5. **보안 위험**: 보안 취약점이 발생할 가능성이 높음

거부 시 반드시 이유를 명확히 문서화해야 합니다.
