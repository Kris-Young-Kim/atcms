# 의사결정 프로세스 가이드

**프로젝트 코드**: ATCMP-2026  
**작성일**: 2025-01-27  
**버전**: 1.0

---

## 📋 개요

이 문서는 AT-CMP 프로젝트에서 기술적 의사결정, 요구사항 변경, 일정 변경, 그리고 승인 프로세스를 체계적으로 관리하기 위한 가이드입니다. 명확한 의사결정 프로세스를 통해 프로젝트의 투명성과 효율성을 향상시킵니다.

---

## 🎯 목적

1. **의사결정의 투명성**: 모든 주요 의사결정이 문서화되고 추적됨
2. **일관성**: 동일한 방식으로 의사결정을 진행
3. **책임 명확화**: 의사결정 권한과 책임이 명확히 정의됨
4. **지식 공유**: 의사결정 배경과 이유가 팀 전체에 공유됨

---

## 📚 의사결정 유형

### 1. 기술 의사결정 (Technical Decision)

**정의**: 기술 스택, 아키텍처, 라이브러리 선택 등 기술적 결정

**범위**:
- 프레임워크 및 라이브러리 선택
- 아키텍처 패턴 선택
- 데이터베이스 스키마 설계
- API 설계 결정
- 인프라 구성 결정

**문서화**: ADR (Architecture Decision Record) 작성 필수

**승인 주체**: 기술 리더 + 관련 개발자 (다수결 또는 합의)

---

### 2. 요구사항 변경 (Requirement Change)

**정의**: 기능 요구사항 추가, 변경, 삭제

**범위**:
- 새로운 기능 추가
- 기존 기능 수정
- 기능 제거
- 사용자 스토리 변경

**문서화**: 변경 요청 이슈 작성 필수

**승인 주체**: PM + 이해관계자 (영향도에 따라 단계별 승인)

---

### 3. 일정 변경 (Schedule Change)

**정의**: 프로젝트 일정, 마일스톤, 스프린트 일정 변경

**범위**:
- 스프린트 일정 변경
- 마일스톤 연기/당기기
- 릴리즈 일정 변경
- 휴가/부재 일정 반영

**문서화**: 일정 변경 요청 이슈 작성 필수

**승인 주체**: PM + 프로젝트 스폰서 (일정 변경 범위에 따라)

---

### 4. 승인 프로세스 (Approval Process)

**정의**: 예산, 리소스, 정책 등에 대한 승인

**범위**:
- 예산 승인
- 외부 서비스 도입 승인
- 보안 정책 변경 승인
- 프로덕션 배포 승인

**문서화**: 승인 요청 문서 작성 필수

**승인 주체**: 승인 권한 매트릭스에 따라 결정

---

## 🔄 의사결정 프로세스

### 단계 1: 의사결정 필요성 인식

1. **문제 또는 기회 식별**
   - 현재 상황 분석
   - 문제점 또는 개선 기회 명확화

2. **의사결정 타입 결정**
   - 기술 의사결정 / 요구사항 변경 / 일정 변경 / 승인

3. **우선순위 평가**
   - 긴급성, 중요도에 따라 우선순위 결정

---

### 단계 2: 정보 수집 및 분석

1. **대안 도출**
   - 가능한 모든 대안 식별
   - 각 대안의 장단점 분석

2. **영향도 분석**
   - 기술적 영향
   - 일정 영향
   - 리소스 영향
   - 위험도 평가

3. **비용/효과 분석**
   - 각 대안의 비용
   - 예상 효과
   - ROI 계산

---

### 단계 3: 의사결정 문서 작성

1. **의사결정 문서 작성**
   - ADR (기술 의사결정)
   - 변경 요청 이슈 (요구사항/일정 변경)
   - 승인 요청 문서 (승인 프로세스)

2. **문서 공유**
   - 관련 이해관계자에게 공유
   - 피드백 수집 기간 확보 (최소 2일)

---

### 단계 4: 검토 및 토론

1. **이해관계자 검토**
   - 기술 리더 리뷰 (기술 의사결정)
   - PM 리뷰 (요구사항/일정 변경)
   - 승인권자 리뷰 (승인 프로세스)

2. **토론 및 질의응답**
   - 기술 회의 (기술 의사결정)
   - 변경 요청 회의 (요구사항/일정 변경)
   - 승인 회의 (승인 프로세스)

---

### 단계 5: 의사결정 및 승인

1. **의사결정 방식 선택**
   - 합의 (Consensus): 모든 이해관계자 동의
   - 다수결 (Majority): 다수 찬성
   - 권한자 결정 (Authority): 승인권자 최종 결정

2. **결정사항 문서화**
   - 결정 내용 기록
   - 결정 이유 명시
   - 대안 선택 이유 설명

3. **승인 완료**
   - 승인 문서 서명 (필요시)
   - 이슈 상태 업데이트
   - 관련 문서 업데이트

---

### 단계 6: 의사결정 추적 및 실행

1. **실행 계획 수립**
   - 작업 항목 생성
   - 담당자 할당
   - 일정 설정

2. **진행 상황 추적**
   - 정기적 진행 상황 확인
   - 이슈 추적

3. **결과 검토**
   - 의사결정 효과 평가
   - 필요시 조정

## 👥 의사결정 권한 체계

### 기술 의사결정

| 결정 범위 | 주도자 | 승인자 | 참여자 |
|---------|--------|--------|--------|
| 프레임워크/라이브러리 선택 | 기술 리더 | 기술 리더 + 개발자 합의 | 전체 개발팀 |
| 아키텍처 설계 | 기술 리더 | 기술 리더 | 관련 개발자 |
| API 설계 | 백엔드 리더 | 기술 리더 | 프론트엔드 리더 |
| 인프라 구성 | 인프라 담당자 | 기술 리더 | DevOps 팀 |

**승인 기준**: 관련 개발자 2/3 이상 동의 또는 기술 리더 최종 결정

---

### 요구사항 변경

| 영향 범위 | 주도자 | 승인자 | 참여자 |
|---------|--------|--------|--------|
| 작은 변경 (< 1일) | 개발자 | PM | 관련 개발자 |
| 중간 변경 (1-3일) | PM | PM + 기술 리더 | 관련 개발자 |
| 큰 변경 (> 3일) | PM | PM + 프로젝트 스폰서 | 전체 팀 |
| 마일스톤 영향 | PM | 프로젝트 스폰서 | 전체 팀 + 이해관계자 |

**승인 기준**: 영향도에 따라 단계별 승인

---

### 일정 변경

| 변경 범위 | 주도자 | 승인자 | 참여자 |
|---------|--------|--------|--------|
| 스프린트 내 조정 | PM | PM | 개발팀 |
| 마일스톤 조정 (< 1주) | PM | PM + 기술 리더 | 전체 팀 |
| 마일스톤 조정 (> 1주) | PM | 프로젝트 스폰서 | 전체 팀 + 이해관계자 |
| 릴리즈 일정 변경 | PM | 프로젝트 스폰서 | 전체 팀 + 이해관계자 |

**승인 기준**: 일정 변경 범위에 따라 단계별 승인

---

### 승인 프로세스

| 승인 항목 | 주도자 | 승인자 | 참여자 |
|---------|--------|--------|--------|
| 예산 (< 100만원) | PM | PM | 재무 담당자 |
| 예산 (> 100만원) | PM | 프로젝트 스폰서 | 재무 담당자 |
| 외부 서비스 도입 | 기술 리더 | 기술 리더 + PM | 보안 담당자 |
| 프로덕션 배포 | DevOps | 기술 리더 + PM | QA 팀 |

**승인 기준**: 승인 권한 매트릭스에 따라 결정

## 🔍 의사결정 품질 기준

### 좋은 의사결정의 특징

1. **명확한 문제 정의**: 무엇을 해결하려는지 명확함
2. **충분한 정보**: 모든 관련 정보가 수집됨
3. **대안 검토**: 여러 대안이 검토됨
4. **영향도 분석**: 영향을 받는 영역이 명확함
5. **문서화**: 결정 사항과 이유가 문서화됨
6. **이해관계자 참여**: 관련 이해관계자가 참여함

## 📊 의사결정 추적

### 추적 지표

- **의사결정 수**: 월별 기술 의사결정 수
- **의사결정 시간**: 의사결정에 소요된 평균 시간
- **의사결정 회수율**: 의사결정 후 변경/회수 비율
- **의사결정 효과**: 의사결정의 목표 달성률

### 리뷰 주기

- **주간**: 진행 중인 의사결정 상태 확인
- **월간**: 의사결정 지표 리뷰
- **분기별**: 의사결정 프로세스 개선 검토
