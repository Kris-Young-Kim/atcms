## 📋 개요

이 문서는 AT-CMP 프로젝트의 커뮤니케이션 채널 및 사용 가이드를 설명합니다. 효율적인 팀 커뮤니케이션을 위한 통합 가이드입니다.

---

## 🎯 커뮤니케이션 목표

1. **명확하고 효율적인 정보 공유**
2. **적절한 채널 활용**
3. **실시간 피드백 및 협업**
4. **지식 축적 및 공유**

---

## 📊 커뮤니케이션 채널

### 1. Slack

**주요 용도**: 실시간 커뮤니케이션, 알림, 빠른 질문 및 답변

**사용 케이스**:
- 일일 스탠드업 (비동기)
- 빠른 질문 및 답변
- 알림 및 공지
- 비공식 논의

**채널 구조**: [Slack 채널 가이드](./slack-channel-guide.md) 참고

**응답 시간 목표**:
- 긴급: 30분 이내
- 일반: 4시간 이내
- 비업무: 24시간 이내

---

### 2. GitHub Issues

**주요 용도**: 작업 추적, 버그 리포트, 기능 요청, 기술 부채 관리

**사용 케이스**:
- 버그 리포트
- 기능 요청
- 작업 할당 및 추적
- 기술 부채 기록
- 이슈 논의

**템플릿**:
- [버그 리포트](../.github/ISSUE_TEMPLATE/bug_report.md)
- [기능 요청](../.github/ISSUE_TEMPLATE/feature_request.md)
- [기술 부채](../.github/ISSUE_TEMPLATE/technical_debt.md)
- [질문](../.github/ISSUE_TEMPLATE/question.md)

**응답 시간 목표**:
- P0: 즉시 확인
- P1: 24시간 이내
- P2: 3일 이내
- P3: 1주일 이내

---

### 3. GitHub Pull Requests

**주요 용도**: 코드 리뷰, 변경사항 논의, 병합 승인

**사용 케이스**:
- 코드 리뷰 요청
- 변경사항 논의
- 병합 승인
- 피드백 제공

**리뷰 시간 목표**:
- 긴급: 2시간 이내
- 일반: 24시간 이내
- 낮은 우선순위: 3일 이내

---

### 4. Google Meet

**주요 용도**: 정기 회의, 화면 공유가 필요한 논의, 복잡한 토픽 논의

**사용 케이스**:
- 일일 스탠드업 (선택사항)
- 주간 계획 회의
- 스프린트 리뷰
- 기술 회의
- 화면 공유가 필요한 논의

**회의실 설정**:
- 정기 회의 링크 생성
- 회의실 예약 시스템 설정 (선택사항)

**회의 규칙**:
- 회의 시작 5분 전 입장
- 회의록 작성 필수
- 액션 아이템 명확히 기록

---

### 5. Notion

**주요 용도**: 문서 관리, 지식 베이스, 프로젝트 대시보드

**사용 케이스**:
- 프로젝트 문서 관리
- 지식 베이스 구축
- 프로젝트 대시보드
- 회의록 보관
- 일일/주간 리포트

**구조**:
- 프로젝트 개요
- 문서 디렉토리
- 회의록 보관소
- 리포트 보관소

## 📝 커뮤니케이션 규칙

### 메시지 작성 규칙

1. **명확한 제목/제목 줄**:
   - 이슈: `[PRIORITY] [TYPE] 제목`
   - PR: `[ATCMP-XXX] 제목`
   - Slack: Thread 사용 권장

2. **컨텍스트 제공**:
   - 배경 정보 포함
   - 관련 링크 제공
   - 스크린샷 또는 로그 첨부

3. **액션 아이템 명시**:
   - 필요한 작업 명확히 표시
   - 담당자 지정
   - 마감일 설정

| GitHub Issues | 즉시 | 24시간 | 3일 |

### 언급 및 알림 규칙

**Slack**:
- `@channel`: 긴급 공지에만 사용
- `@here`: 현재 온라인인 팀원에게만
- `@username`: 특정 사용자에게

**GitHub**:
- `@username`: 특정 사용자에게 알림
- `@team-name`: 팀 전체에게 알림 (예: @atcmp-team)

## 🔄 커뮤니케이션 플로우

### 버그 리포트 플로우

1. **발견**: 사용자 또는 개발자가 버그 발견
2. **보고**: GitHub Issues에 버그 리포트 작성
3. **알림**: Slack `#alerts` 채널에 자동 알림
4. **할당**: 담당자 지정 및 우선순위 설정
5. **수정**: 개발자가 수정 작업 진행
6. **완료**: 버그 수정 완료 및 이슈 종료

### 기능 요청 플로우

1. **제안**: GitHub Issues에 기능 요청 작성
2. **논의**: 팀 회의에서 논의 및 우선순위 결정
3. **계획**: 스프린트 계획에 포함
4. **구현**: 개발 작업 진행
5. **완료**: 기능 구현 완료 및 이슈 종료

### 코드 리뷰 플로우

1. **PR 생성**: 개발자가 Pull Request 생성
2. **리뷰 요청**: 리뷰어 지정 및 알림
3. **리뷰**: 코드 리뷰 및 피드백 제공
4. **수정**: 피드백 반영 및 재리뷰
5. **승인**: 리뷰 승인 및 병합

## 📋 채널별 사용 가이드

### Slack 사용 가이드

**언제 사용하는가?**
- 빠른 질문 및 답변
- 실시간 알림
- 비공식 논의
- 일일 스탠드업 (비동기)

**언제 사용하지 않는가?**
- 복잡한 기술 논의 (GitHub Issues 또는 회의 사용)
- 중요한 결정사항 (회의 또는 GitHub Issues 사용)
- 장기 보관이 필요한 정보 (Notion 사용)

**자세한 내용**: [Slack 채널 가이드](./slack-channel-guide.md)

---

### GitHub Issues 사용 가이드

**언제 사용하는가?**
- 버그 리포트
- 기능 요청
- 작업 추적
- 기술 부채 기록
- 장기 논의

**언제 사용하지 않는가?**
- 빠른 질문 (Slack 사용)
- 코드 리뷰 (Pull Request 사용)

**자세한 내용**: [이슈 관리 가이드](./issue-management-guide.md)

---

### Google Meet 사용 가이드

**언제 사용하는가?**
- 정기 회의 (주간 계획, 스프린트 리뷰)
- 화면 공유가 필요한 논의
- 복잡한 토픽 논의
- 기술 회의

**언제 사용하지 않는가?**
- 간단한 질문 (Slack 사용)
- 단방향 공지 (Slack 또는 Email 사용)

**회의 규칙**:
- 회의 시작 5분 전 입장
- 회의록 작성 필수
- 액션 아이템 명확히 기록

---

### Notion 사용 가이드

**언제 사용하는가?**
- 프로젝트 문서 관리
- 지식 베이스 구축
- 회의록 보관
- 일일/주간 리포트
- 장기 보관이 필요한 정보

**언제 사용하지 않는가?**
- 실시간 논의 (Slack 사용)
- 작업 추적 (GitHub Issues 사용)
