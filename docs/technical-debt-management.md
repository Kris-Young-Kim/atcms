## 📊 기술 부채 분류

### 1. 우선순위 (Priority)

**P0 (긴급)**: 즉시 해결 필요
- 보안 취약점
- 프로덕션 장애 위험
- 데이터 손실 위험

**P1 (높음)**: 1-2주 내 해결
- 성능 저하
- 사용자 경험 저해
- 확장성 문제

**P2 (중간)**: 1-2개월 내 해결
- 코드 품질 개선
- 리팩토링 필요
- 문서화 부족

**P3 (낮음)**: 계획적으로 해결
- 코드 스타일 개선
- 마이너 리팩토링
- 선택적 개선

### 2. 유형 (Type)

**아키텍처 부채**:
- 잘못된 설계 결정
- 기술 스택 선택 문제
- 시스템 구조 문제

**코드 부채**:
- 중복 코드
- 복잡한 로직
- 낮은 테스트 커버리지

**문서 부채**:
- 문서화 부족
- 오래된 문서
- API 문서 부재

**의존성 부채**:
- 오래된 라이브러리
- 보안 취약점 포함
- 유지보수 중단된 라이브러리

**인프라 부채**:
- 오래된 인프라
- 비용 최적화 필요
- 모니터링 부족

## 🔍 기술 부채 추적 방법

### 1. GitHub Issues 사용

**라벨 구조**:
- `technical-debt`: 기술 부채 이슈
- `tech-debt:P0`, `tech-debt:P1`, `tech-debt:P2`, `tech-debt:P3`: 우선순위
- `tech-debt:architecture`: 아키텍처 부채
- `tech-debt:code`: 코드 부채
- `tech-debt:documentation`: 문서 부채
- `tech-debt:dependency`: 의존성 부채
- `tech-debt:infrastructure`: 인프라 부채

**이슈 템플릿**: `docs/issue-templates/technical-debt.md`

### 2. 기술 부채 레지스트리

**파일**: `docs/technical-debt-registry.md`

기술 부채를 중앙에서 관리하는 레지스트리입니다.

**포맷**:
```markdown
## [우선순위] [유형] 제목

**발견일**: YYYY-MM-DD  
**담당자**: @username  
**상태**: Open | In Progress | Resolved  

**설명**:
기술 부채에 대한 상세 설명

**영향**:
- 성능 저하
- 유지보수 어려움

**해결 방안**:
해결 방법 및 예상 작업

**참고**:
관련 이슈 또는 PR 링크
```

### 3. 코드 주석으로 표시

**TODO 주석 형식**:
```typescript
// TODO(TECH-DEBT): [우선순위] 설명
// 예: TODO(TECH-DEBT): [P1] 이 함수는 리팩토링 필요
```

**FIXME 주석 형식**:
```typescript
// FIXME(TECH-DEBT): [우선순위] 설명
// 예: FIXME(TECH-DEBT): [P0] 보안 취약점 수정 필요
```

## 📈 기술 부채 측정 지표

### 1. 코드 메트릭

**복잡도 (Cyclomatic Complexity)**:
- 목표: 함수당 10 이하
- 경고: 15 초과
- 위험: 25 초과

**코드 중복**:
- 목표: 0%
- 허용: 5% 이하
- 개선 필요: 10% 초과

**테스트 커버리지**:
- 목표: 70% 이상
- 허용: 50% 이상
- 개선 필요: 50% 미만

### 2. 의존성 메트릭

**오래된 패키지**:
- 목표: 모든 패키지 최신 버전
- 경고: 6개월 이상 업데이트 없음
- 위험: 1년 이상 업데이트 없음

**보안 취약점**:
- 목표: 0개
- 허용: 중간 이하 5개 이하
- 위험: 높음 이상 또는 중간 이상 10개

### 3. 문서화 메트릭

**API 문서화율**:
- 목표: 100%
- 허용: 80% 이상
- 개선 필요: 80% 미만

**컴포넌트 문서화율**:
- 목표: 100%
- 허용: 70% 이상
- 개선 필요: 70% 미만

## 🔄 기술 부채 관리 프로세스

### 1. 발견 및 기록

**발견 시점**:
- 코드 리뷰 중
- 리팩토링 중
- 버그 수정 중
- 새로운 기능 개발 중

**기록 방법**:
1. GitHub Issue 생성
2. 기술 부채 레지스트리에 추가
3. 코드에 TODO 주석 추가 (선택)

### 2. 우선순위 설정

**평가 기준**:
- **영향도**: 높음, 중간, 낮음
- **긴급도**: 긴급, 높음, 중간, 낮음
- **복잡도**: 높음, 중간, 낮음

**우선순위 공식**:
```
우선순위 = 영향도 × 긴급도 × 복잡도
```

### 3. 정기 리뷰

**주간 리뷰**:
- 새로 발견된 기술 부채 검토
- 우선순위 재평가
- 진행 중인 기술 부채 상태 확인

**월간 리뷰**:
- 기술 부채 전체 현황 분석
- 해결 계획 수립
- 리스크 평가

**분기별 리뷰**:
- 기술 부채 전략 수립
- 장기 계획 수정
- 팀 회의에서 공유

### 4. 해결 계획

**스프린트 할당**:
- 각 스프린트에 기술 부채 작업 10-20% 할당
- P0 기술 부채는 즉시 해결
- P1 기술 부채는 다음 스프린트에 포함

**해결 전략**:
- **점진적 개선**: 큰 변경을 작은 단계로 분할
- **리팩토링**: 점진적으로 코드 품질 개선
- **교체**: 오래된 기술을 새로운 기술로 교체
