# 위험 평가 매트릭스

**프로젝트 코드**: ATCMP-2026  
**작성일**: 2025-01-27  
**버전**: 1.0

---

## 📋 개요

이 문서는 프로젝트에서 발생할 수 있는 위험을 체계적으로 평가하기 위한 매트릭스를 정의합니다. 영향도와 확률을 기반으로 위험 점수를 계산하고 우선순위를 결정합니다.

---

## 🎯 목적

1. **위험 평가의 객관성**: 일관된 기준으로 위험을 평가
2. **우선순위 결정**: 위험 점수를 기반으로 우선순위 결정
3. **자원 배분**: 위험 등급에 따라 적절한 자원 배분
4. **대응 전략 수립**: 위험 등급에 따라 적절한 대응 전략 수립

---

## 📊 영향도 척도 (Impact Scale)

### 영향도 정의

| 등급 | 점수 | 정의 | 설명 | 예시 |
|------|------|------|------|------|
| 🔴 **매우 높음** | 5 | 프로젝트 실패 | 프로젝트 목표 달성 불가능 | 핵심 인프라 장애, 주요 개발자 이탈 |
| 🟠 **높음** | 4 | 주요 마일스톤 실패 | 일부 목표 달성 어려움 | 주요 기능 지연, 예산 초과 |
| 🟡 **중간** | 3 | 기능 지연 또는 품질 저하 | 일정 지연 또는 품질 저하 발생 | 중간 기능 지연, 테스트 커버리지 감소 |
| 🟢 **낮음** | 2 | 작은 지연 또는 경미한 품질 저하 | 프로젝트 목표 달성 가능 | 사소한 버그, 문서화 지연 |
| ⚪ **매우 낮음** | 1 | 미미한 영향 | 거의 영향 없음 | 스타일 변경, 작은 UI 개선 지연 |

### 영향도 평가 기준

#### 기술적 영향

- **매우 높음**: 핵심 시스템 장애, 데이터 손실 위험
- **높음**: 주요 기능 사용 불가, 보안 취약점
- **중간**: 일부 기능 제한, 성능 저하
- **낮음**: 사소한 기능 제한, UI 문제
- **매우 낮음**: 거의 영향 없음

#### 일정 영향

- **매우 높음**: 릴리즈 일정 2주 이상 지연
- **높음**: 마일스톤 1주 이상 지연
- **중간**: 스프린트 3일 이상 지연
- **낮음**: 작업 1일 지연
- **매우 낮음**: 거의 영향 없음

#### 비용 영향

- **매우 높음**: 예산 초과 20% 이상
- **높음**: 예산 초과 10-20%
- **중간**: 예산 초과 5-10%
- **낮음**: 예산 초과 5% 미만
- **매우 낮음**: 거의 영향 없음

#### 품질 영향

- **매우 높음**: 심각한 보안 취약점, 데이터 무결성 문제
- **높음**: 주요 기능 오작동, 사용자 불만 증가
- **중간**: 일부 기능 오작동, 사용성 저하
- **낮음**: 사소한 버그, UI 개선 필요
- **매우 낮음**: 거의 영향 없음

## 📈 확률 척도 (Probability Scale)

### 확률 정의

| 등급 | 점수 | 정의 | 설명 | 예시 |
|------|------|------|------|------|
| 🔴 **매우 높음** | 5 | 80% 이상 | 거의 확실히 발생 | 외부 API 장애 이력, 계약 만료 임박 |
| 🟠 **높음** | 4 | 60-80% | 발생 가능성이 높음 | 기술 스택 호환성 문제, 팀원 이탈 가능성 |
| 🟡 **중간** | 3 | 40-60% | 발생 가능성이 보통 | 일정 지연 가능성, 예상 작업량 초과 |
| 🟢 **낮음** | 2 | 20-40% | 발생 가능성이 낮음 | 외부 서비스 지연, 작은 기술적 문제 |
| ⚪ **매우 낮음** | 1 | 20% 미만 | 발생 가능성이 매우 낮음 | 예상치 못한 상황, 드문 이벤트 |

### 확률 평가 기준

#### 과거 데이터 기반

- **매우 높음**: 과거에 빈번히 발생 (80% 이상 발생률)
- **높음**: 과거에 자주 발생 (60-80% 발생률)
- **중간**: 과거에 가끔 발생 (40-60% 발생률)
- **낮음**: 과거에 드물게 발생 (20-40% 발생률)
- **매우 낮음**: 과거에 거의 발생하지 않음 (20% 미만 발생률)

#### 상황 분석 기반

- **매우 높음**: 현재 상황에서 발생 가능성이 매우 높음
- **높음**: 현재 상황에서 발생 가능성이 높음
- **중간**: 현재 상황에서 발생 가능성이 보통
- **낮음**: 현재 상황에서 발생 가능성이 낮음
- **매우 낮음**: 현재 상황에서 발생 가능성이 매우 낮음

## 🎯 위험 등급 매트릭스

### 위험 점수 계산

**위험 점수** = 영향도 점수 × 확률 점수

### 위험 등급 매트릭스

| 영향도 ↓ / 확률 → | 매우 낮음 (1) | 낮음 (2) | 중간 (3) | 높음 (4) | 매우 높음 (5) |
|------------------|-------------|---------|---------|---------|-------------|
| **매우 높음 (5)** | 5 🟡 P2 | 10 🔴 P0 | 15 🔴 P0 | 20 🔴 P0 | 25 🔴 P0 |
| **높음 (4)** | 4 🟡 P2 | 8 🟠 P1 | 12 🔴 P0 | 16 🔴 P0 | 20 🔴 P0 |
| **중간 (3)** | 3 🟢 P3 | 6 🟠 P1 | 9 🔴 P0 | 12 🔴 P0 | 15 🔴 P0 |
| **낮음 (2)** | 2 🟢 P3 | 4 🟡 P2 | 6 🟠 P1 | 8 🟠 P1 | 10 🔴 P0 |
| **매우 낮음 (1)** | 1 🟢 P3 | 2 🟢 P3 | 3 🟢 P3 | 4 🟡 P2 | 5 🟡 P2 |

### 위험 등급 및 우선순위

| 위험 점수 | 위험 등급 | 우선순위 | 대응 전략 | 모니터링 주기 |
|----------|----------|----------|----------|------------|
| **16-25** | 🔴 매우 높음 | P0 (긴급) | 즉시 대응 필수 | 일일 |
| **10-15** | 🔴 높음 | P0 (긴급) | 즉시 대응 권장 | 일일 |
| **8-9** | 🟠 높음 | P1 (우선) | 우선 대응 | 주 2회 |
| **6-7** | 🟠 중간 | P1 (우선) | 계획적 대응 | 주 1회 |
| **4-5** | 🟡 중간 | P2 (모니터링) | 모니터링 후 대응 | 주 1회 |
| **3** | 🟡 낮음 | P2 (모니터링) | 정기 검토 | 월 1회 |
| **1-2** | 🟢 낮음 | P3 (정기 검토) | 필요시 대응 | 월 1회 |

## 📋 위험 평가 프로세스

### 단계 1: 위험 식별

1. **위험 발견**
   - 팀원이 위험 발견
   - 프로젝트 리뷰 중 발견
   - 외부 요인 변화 감지

2. **위험 기록**
   - 위험 로그 템플릿 작성
   - 위험 정보 입력

---

### 단계 2: 영향도 평가

1. **영향 범위 분석**
   - 기술적 영향 분석
   - 일정 영향 분석
   - 비용 영향 분석
   - 품질 영향 분석

2. **영향도 점수 결정**
   - 영향도 척도에 따라 점수 결정
   - 근거 문서화

---

### 단계 3: 확률 평가

1. **발생 가능성 분석**
   - 과거 데이터 분석
   - 현재 상황 분석
   - 관련 요인 분석

2. **확률 점수 결정**
   - 확률 척도에 따라 점수 결정
   - 근거 문서화

---

### 단계 4: 위험 점수 계산 및 등급 결정

1. **위험 점수 계산**
   - 영향도 점수 × 확률 점수
   - 위험 점수 기록

2. **위험 등급 결정**
   - 위험 등급 매트릭스 참조
   - 우선순위 결정

---

### 단계 5: 대응 전략 수립

1. **대응 전략 선택**
   - 위험 등급에 따라 대응 전략 선택
   - 완화 전략 수립

2. **모니터링 계획**
   - 모니터링 주기 결정
   - 모니터링 담당자 지정

---

## 📊 위험 평가 예시

### 예시 1: 외부 API 장애

**영향도**: 높음 (4)
- 기술적 영향: 핵심 기능 사용 불가
- 일정 영향: 마일스톤 지연 가능

**확률**: 높음 (4)
- 과거에 장애 이력 있음
- 외부 서비스 안정성 불확실

**위험 점수**: 4 × 4 = 16
**위험 등급**: 🔴 매우 높음
**우선순위**: P0 (긴급)

---

### 예시 2: 개발자 이탈

**영향도**: 높음 (4)
- 기술적 영향: 주요 기능 개발 지연
- 일정 영향: 마일스톤 지연 가능

**확률**: 중간 (3)
- 계약 만료 예정 있음
- 이탈 가능성 있음

**위험 점수**: 4 × 3 = 12
**위험 등급**: 🔴 높음
**우선순위**: P0 (긴급)

---

### 예시 3: 기술 스택 호환성 문제

**영향도**: 중간 (3)
- 기술적 영향: 추가 개발 시간 필요
- 일정 영향: 스프린트 지연 가능

**확률**: 중간 (3)
- 기술 스택 문서화 부족
- 호환성 확인 필요

**위험 점수**: 3 × 3 = 9
**위험 등급**: 🟠 높음
**우선순위**: P1 (우선)

## 🔍 위험 재평가

### 재평가 시점

- **정기 재평가**: 월 1회
- **상황 변화 시**: 즉시 재평가
- **완화 조치 후**: 완화 조치 효과 평가

### 재평가 프로세스

1. **현재 상황 재분석**
   - 영향도 재평가
   - 확률 재평가

2. **위험 점수 재계산**
   - 새로운 위험 점수 계산
   - 위험 등급 재결정

3. **대응 전략 조정**
   - 필요시 대응 전략 조정
   - 모니터링 주기 조정
